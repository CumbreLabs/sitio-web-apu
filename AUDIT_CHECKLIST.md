# Audit Checklist

A reusable framework for auditing this site. Every category below is a real angle that has surfaced bugs in this codebase at some point. Use it to systematically rule out issues instead of doing scattershot reviews.

**Legend**

- ✅ Verified clean in recent audit rounds
- ⚠️ Known concern, partial fix or accepted trade-off
- ❌ Not yet audited — fresh angle to investigate
- 🚫 Permanently blocked or N/A — skip in future audit runs; the rationale is in the row
- 🛠 Tool/command that helps verify

"date-desc"` / `!== false` were scattered across `WritingsTemplate`, `PortfolioTemplate`, `CollectionFeatured*`, `SEO.svelte`, and `routes/sitemap.xml/+server.ts` — changing a default required hunting through `.svelte` files. The contract now lives in one auditable place, and `check:required-config` makes "I forgot to set this" a build-time error rather than a runtime ugly-string in production |

## How to use this checklist

**Before each release**: walk sections 1, 5, 6, 8, 9, 13, 18 — these catch real issues most often.

**Quarterly deep audit**: re-walk every category row-by-row, update the "Last audited" date in the table at the top, do everything marked ❌. Write a 5-line summary of new findings into the most recent section's notes.

**After adding a feature**: spot-check sections relevant to what changed:

- new image flow → 5, 16
- new component → 3, 8, 14, 18.1-18.2 (`npm run check:fork`)
- new config field → 17, 18.6
- new template/section → 9, 10, 11, 18.3-18.4
- new third-party dep → 2, 6, 7, 12

**When the user reports "the site feels slow / wrong"**: start at 5 (Web Vitals) and 11 (cross-browser).

**When the user reports "I forked this and X broke"**: start at 18 (Forking Readiness).

When you fix one of the ❌ items, mark it ✅ here, add the date to the section's "Last audited" row, and link the commit. The checklist is a living document.

## Backlog — suspected fragilities to investigate

These are not current bugs, but plausible angles where a regression could land without anyone noticing. Promote to ❌ rows in the right section when a maintainer commits to investigating, or close out as 🚫 if the angle turns out to be moot.

- **Sharp upgrades silently change WebP encoding parameters.** When sharp@0.34 → 0.35 (or similar), the `quality:80` flag may map to different bits-per-pixel. Bundle-size budget (2.9) catches massive shifts but not small drift. Worth a perceptual diff before each sharp bump.
- **Sveltia CMS minor bumps deprecate widget types or change field-rendering behavior.** We pin to `0.167.3` with SRI (row 6.9), and there's no automated upgrade gate — Sveltia gets bumped manually. Add a smoke test: load `/admin/` headlessly, assert each of the 7 config files' form renders without console errors.
- **GitHub Pages SLA / CDN edge stale-cache behavior.** Long-tail CDN edges may serve old HTML for hours after a deploy. Visible only via "I just deployed, why does this country still see the old page?" Easy diagnostic: `curl -I https://goanpeca.co/<path>` from a few regions.
- **`virtual:image-dims` cache invalidation.** The sharp manifest measures every image in `static/media/` at build start. If a maintainer replaces a file with the same name but different dimensions WITHOUT touching mtime (rare but possible via git), the manifest stays stale and `<img>` tags emit wrong width/height → CLS. `optimize-images.mjs` mtime check would notice.
- **`document.startViewTransition` polyfill assumption.** If page transitions are ever added (currently not used), Safari ≤16 will silently skip the animation. Listed as row 11.10 — keep in this backlog until used.
- **`languages: ["en","es","fr"]` sitemap pairing.** `getEquivalentPath()` and the hreflang emitter pair EN ↔ ES by id. A third language hasn't been tested; album/post hreflang would need triple-pairing. Row 18.9 plus a sitemap visual diff.
